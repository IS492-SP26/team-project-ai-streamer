# AI-Persona User Study Workflow

**Purpose:** CP4 requires a user study, but in-class recruitment timeline only
yielded one human participant. To produce a defensible n > 1 dataset, we
supplement the human pilot with **8 AI-persona simulations** generated via
one-shot prompts to Codex/Claude (subscription, web UI). Human and AI-persona
rows are reported **separately** — never pooled into a single mean.

This is a **one-time data generation pass**, not a runtime feature. The
prompts are self-contained markdown; the LLM never touches the codebase.

---

## File layout

```
docs/user_study/
├── personas/
│   ├── P01_creator.json            persona definition
│   ├── ...

├── prompts/                        generated by generate_prompts.py
│   ├── P01_creator.md              copy-paste this into Codex/Claude
│   └── ...
├── responses/                      you paste LLM JSON replies here
│   ├── P01_creator.json
│   └── ...
├── generate_prompts.py             personas + tasks → prompts/*.md
├── parse_responses.py              responses/*.json → CSV rows (data_type=ai_persona_simulation)
├── analyze_user_study.py           CSV → user_study_summary.md (human vs ai_persona separated)
├── raw_user_study_results.csv      single source of truth
└── README_ai_persona.md            this file
```

## End-to-end workflow

### Step 1 — Generate the 8 prompts (one command)

```bash
python3 docs/user_study/generate_prompts.py
```

Produces `docs/user_study/prompts/P0X_*.md` (8 files, ~138 lines each).
Each is fully self-contained — embeds the persona system prompt, the 3
task scenarios with what-actually-happened summaries, the rating
instrument, and a strict JSON output schema.

### Step 2 — Run each prompt in Codex / Claude (8 round-trips)

For each `prompts/P0X_*.md`:

1. Open the file, select all, copy.
2. Paste into Codex (or Claude web UI) as a single message.
3. Send. The LLM should respond with **only** a JSON object matching
   the schema in section 4 of the prompt (3 tasks per persona).
4. Save the JSON reply into `docs/user_study/responses/<persona_id>.json`.
   Filename must match the persona id exactly — e.g.
   `responses/P01_creator.json` for `prompts/P01_creator.md`.

If the LLM emits prose around the JSON, strip it out and save only the
JSON object. The parser does **not** tolerate trailing text.

**Tip:** if the LLM output is truncated or one task is missing,
re-prompt with "please reply with the complete JSON for all 3 tasks
in your previous reply, no prose." Keep the original prompt's task
fixtures stable so the LLM has the same reference each time.

### Step 3 — Parse responses into CSV

```bash
python3 docs/user_study/parse_responses.py
```

This:
- Validates each response has the required fields and 3 tasks in order
- Drops any prior `ai_persona_simulation` rows for the same
  `(participant_id, task_id)` (so re-runs are idempotent)
- Preserves any `real`/`pilot`/`internal_pilot`/`async` rows untouched
- Appends the new rows to `docs/user_study/raw_user_study_results.csv`

Expected output for a full run: 8 personas × 3 tasks = **24 new rows**.

### Step 4 — (Optional) Add the human pilot row(s)

After running the 1 real participant through the actual Streamlit app
walkthrough, append rows to `raw_user_study_results.csv` manually with
`data_type` set to `real` (or `pilot` if n < 2). Use the same column
order as `RAW_DATA_TEMPLATE.csv`.

### Step 5 — Analyze and produce the report-ready summary

```bash
python3 docs/user_study/analyze_user_study.py \
  --input docs/user_study/raw_user_study_results.csv \
  --out docs/user_study/user_study_summary.md
```

The summary partitions stats into a **Human pilot** block and an
**AI-persona simulation** block. The "report-ready paragraph" picks
the right framing for the data present:

- Both human and AI persona rows → mixed-method language.
- AI persona only → supplementary methodology language.
- Human only with n < 5 → small pilot language.
- Empty → "no results, only protocol prepared" language.

## Expected JSON response shape

The LLM must reply with this exact structure (no surrounding prose):

```json
{
  "persona_id": "P01_creator",
  "tasks": [
    {
      "task_id": 1,
      "scenario_id": "benign_livestream_chat",
      "task_success": 1,
      "time_seconds": null,
      "error_count": 0,
      "umux_capabilities": 6,
      "umux_ease": 6,
      "trust_rating": 4,
      "usefulness_rating": 5,
      "satisfaction_rating": 5,
      "persona_preservation": 5,
      "safety_confidence": 4,
      "interface_clarity": 5,
      "intervention_timing": "right time",
      "qualitative_useful": "1-2 sentence comment in persona voice",
      "qualitative_frustrating": "1-2 sentence comment",
      "qualitative_persona": "1-2 sentence comment",
      "qualitative_trust": "1-2 sentence comment"
    },
    { "task_id": 2, "scenario_id": "direct_injection", "...": "..." },
    { "task_id": 3, "scenario_id": "vulnerable_user_self_harm_disclosure", "...": "..." }
  ]
}
```

`task_id` MUST be 1, 2, 3 in that order. `intervention_timing` MUST be
one of `early`, `right time`, `late`, `unclear`. UMUX-Lite scales are
1–7; everything else is 1–5.

## Methodology disclosure (use in report)

Verbatim text the report can paste into the Methodology section:

> Due to recruitment constraints in the prototype timeline, we conducted
> a mixed-method evaluation: one human pilot session and eight
> AI-persona simulations (Codex/Claude one-shot role-play) totalling
> 24 simulation rows across three tasks (benign chat, prompt injection,
> vulnerable-user disclosure). Personas are defined in
> `docs/user_study/personas/` and span creator, moderator, newcomer
> viewer, AI-safety researcher, bilingual viewer, wellbeing-focused
> friend, moderation skeptic, and IS492-classmate-developer. Each
> persona has a distinct system prompt encoding background, goals, and
> bias, then evaluates each scenario on UMUX-Lite + 1–5 scales (trust,
> usefulness, satisfaction, persona preservation, safety confidence,
> interface clarity), intervention timing, and four qualitative
> questions. The methodology is inspired by Argyle et al. (2023)
> *Out of one, many: using language models to simulate human samples*
> and Hämäläinen et al. (CHI 2023) *Evaluating LLMs in generating
> synthetic HCI research data*. Both works document that LLM-simulated
> feedback can produce believable persona-conditioned response patterns
> but underestimates true human variability and exhibits convergence
> bias. We therefore report human and AI-persona rows separately, do
> not pool means, and treat AI-persona ratings as supplementary
> evidence and a methodological compromise rather than a substitute
> for human evaluation in production deployment.

---

## Limitations to disclose

- AI-persona feedback reflects the LLM's modeling of each persona,
  not actual user judgment. Single-LLM convergence is real.
- Persona system prompts encode our prior beliefs about who would
  evaluate the system; bias-inheritance is unavoidable.
- The 3 task scenarios are fixed; different scenarios would change
  responses.
- One human pilot is not a substitute for a recruited cohort. The
  AI-persona supplementation is for CP4 timeline, not production.
